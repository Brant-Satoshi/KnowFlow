# ADR-011：检索为空时由代码拒答；rerank 分数下限不采用（默认 0）

## 状态
已接受

## 背景
在此之前，chat 管线对每一个问题都会调用 LLM——**包括召回为 0 条 chunk 的问题**。
此时是否拒答，完全取决于 `buildQaPrompt` 里那句 "If the answer cannot be found in the
context, say exactly ..."。那是一个**请求**，不是**保证**：空上下文恰恰是模型最容易
用自身权重去填补沉默的场合，而这正是"知识库里没有的答案"被编造出来的地方。

需要一个决策，是因为"不知道时不要乱答"有两种可能的实现，而它们的可靠性差别很大：

1. **召回为空 → 代码拒答**（确定性，不依赖模型自觉）
2. **召回分数过低 → 代码拒答**（需要一个可信的"相关性下限"）

第 2 条听起来是第 1 条的自然延伸，也是任务清单里明确列出的一项。但它成立的前提是
**rerank 相关性分数能够区分"可回答"和"不可回答"**。这个前提必须被度量，而不是被假设。

## 决策

### 采用：召回为空即拒答
`assessRetrieval()`（`lib/rag/refusal-gate.ts`）在 `selectFinalChunks` 之后运行。
最终 chunk 为空时，`emitRefusal()`（`lib/llm/refusal.ts`）把固定拒答文案当作一次
普通回答流式发出（`meta → progress → 单个 token → done`）并落库，**完全不调用 LLM**。

- `meta.refusal` 携带机器可读的原因（`'empty' | 'low_score'`）。这是**唯一**能证明闸门
  生效的信号：拒答文案本身证明不了任何事——prompt 本来就要求 LLM 输出同一句话。
- 唯一豁免是**对话摘要**（`isConversationSummaryQuery`）：它本就该从历史而非检索作答。
  为此收窄了 `isSummaryQuery` 的用法——它是宽松子串匹配，原先会把
  "Summarize information about X" 在零召回时路由到 `buildConversationSummaryPrompt`，
  **丢弃用户的问题**转而总结聊天记录，等于回答了一个用户没问的问题。

### 不采用：rerank 分数下限（`RETRIEVAL.minRerankScore = 0`，机制保留但失效）

用 `pnpm eval:refusal` 实测 `cohere/rerank-v3.5` 的分数分布：每语言 7 条负样本 + 9 条
正样本，**EN 校准 / ZH 留出验证**，每个数据集跑在只装了它自己那份 fixture 的 KB 上
（混装双语 KB 会让英文负样本从中文文档里被答出来——这个坑我们踩过，见下）。

| | 最低分的**可回答**问题 | 最高分的**不可回答**问题 |
| --- | ---: | ---: |
| olympus (EN) | 0.8808（"41 小时压力波是哪座站发现的？"） | **0.9055**（"是哪座站记录到塔尔西斯 18% 尘暴增强？"） |
| olympus-zh (ZH) | 0.8212 | 0.8003 |

**EN 上分布是重叠的，而且方向相反**：reranker 给"文档回答不了的问题"的分数，高于它给
"文档明确回答得了的问题"的分数。原因不难理解——cross-encoder 衡量的是
**"这段 chunk 和这个 query 是不是在谈同一件事"**，而近似负样本恰恰在谈同一件事，
它只是问了文档唯一没写的那个事实。**相关性 ≠ 可回答性**，分数里不携带后者的信息。

校准脚本在 EN 上确实能找到一个"零新增误拒"的下限 **0.875**——但它之所以成立，只是因为
它恰好卡在那条 0.8808 的**可回答**问题下方 **0.0058**。9 条正样本里的最低分是**样本最小值**，
不是总体最小值。这不是校准出来的阈值，是运气。

**留出集当场证明了这一点**：把 EN 选出的 0.875 拿到没参与选阈值的 ZH 负样本上跑，
out-of-scope 拒答率 100%，但 **in-scope 误拒率 11.1%**——它拒掉了
`olympus-zh-relay-satellite`（0.8212），一个文档明明白白答得出来的问题。
阈值拟合的是 EN 那一批样本，不是 reranker。

因此：**下限机制保留并测试完备，但默认 0（失效）**。它同时在以下情况自动失效，
因为此时它无法被信任：rerank 关闭、rerank 降级为召回序、单 chunk 短路（拿不到分数）、
或配置了非校准过的 reranker（`OPENROUTER_RERANK_MODEL` ≠ `cohere/rerank-v3.5`——
分数在不同 reranker 之间不可比）。

**近似负样本该由谁兜底**：由 prompt——而且实测它做得到。把两个数据集全部 14 条负样本
走完整管线（召回 → rerank → 闸门 → LLM）：**14/14 全部拒答**，零编造。其中 4 条由代码闸门
（空召回）拦下，10 条由 QA prompt 拦下——包括最凶的 0.8652 near-miss（"第二阶段预算是多少"），
模型明确回答"上下文里没有说明"。

这不是巧合：模型能**看见 chunk 的内容**，因而能判断"这个事实不在里面"；代码只能看见一个
相似度标量，看不见这件事。两种机制是互补的，不是替代关系：代码负责"什么都没检索到"，
prompt 负责"检索到了但答案不在里面"。硬塞一个下限在中间，是在一个不携带该信息的信号上做决策。

## 备选方案

### 直接上线 0.875 的下限
优点：
- EN 校准集上把 out-of-scope 拒答率从 28.6% 提到 71.4%，且在该集上零新增误拒

缺点：
- **留出集直接否掉了它**：同一个阈值在 ZH 上误拒 11.1% 的可回答问题
- 距离一条**真实可回答**问题只有 0.0058 的余量；语料重切分、换 embedding、甚至换一批用户提问，
  都会让它开始拒答答得出来的问题——而误拒是比"偶尔答错"更难被用户原谅的失败
- 它要解决的问题（近似负样本编造）实测**根本不存在**：prompt 已经 14/14 全拒了

### 用 LLM judge 判断"上下文是否足以回答"再决定拒答
优点：
- 这是唯一真正能捕捉"可回答性"的信号

缺点：
- 每个请求多一次 LLM 往返（延迟 + 成本），而 QA prompt 本就在做同一件判断——
  等于花钱把已经在做的事再做一遍
- 若要做，应先用 eval 证明 QA prompt 的近似负样本拒答率确实不达标

## 后果
- 空召回不再可能产生编造答案，且省掉一次 LLM 调用（拒答是秒回的）。
- **负样本必须对着整份 fixture 核实过再写。** 本 ADR 的第一版把"是哪座站记录到塔尔西斯 18%"
  当成中文的负样本——而 `sample-zh.txt` 第 113 行白纸黑字写着"赫利俄斯首先发现塔尔西斯东部
  尘埃快速增加"。模型带引用正确回答了它，却被记成了幻觉。**grep 整份文档，别只扫一眼。**
  同理，每个数据集必须跑在只装它自己 fixture 的 KB 上：在双语 KB 里，英文负样本会被中文文档答出来。
- `oosRefusalRate` / `inScopeFalseRefusalRate`（`lib/eval/runner.ts`）成为衡量拒答正确性的指标。
  原有的 `retrievalHit` / `citationHit` **结构上无法**衡量它：out-of-scope 用例没有 target chunk，
  `retrievalHit`（"没检索到相关内容"）无论系统是拒答还是编造都恒为真；`citationHit`（"没有引用"）
  对"没引用的胡说"和"拒答"一视同仁地判过。
- 重新评估下限前必须重跑 `pnpm eval:refusal`，并且**换 reranker 后阈值一律作废**。
- 已知边界：召回恰好 1 条 chunk 时 `rerankChunks` 短路，该 chunk 没有分数，下限天然失效
  （仍受召回阶段 `maxDistance 0.6` 保护）。实测中 `olympus-oos-relay-latency` 正是这种情况。
