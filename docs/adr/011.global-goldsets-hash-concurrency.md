# ADR-011：评测集全局归属 + revision 乐观并发（hash 作内容标识）

## 状态

已接受

## 背景

评测用例集（goldset）原先硬编码在 `lib/eval/dataset.ts`，只能改代码扩展；`eval_datasets` / `eval_cases` 表已存在（006），但仅作为运行时由 `ensureDataset` 写入的代码快照。要让用户在 `/eval` 页面管理用例（表单 + JSON 批量导入），必须决定两件事：goldset 挂在哪一级租户下，以及并发编辑与"run 结果是否可比"的一致性模型。

约束：eval run 本身是 KB 范围的（`requireKnowledgeBaseAccess`）；`eval_runs` 已有 `dataset_name` / `dataset_hash` 快照列与 `eval_runs_hash_idx`；删 dataset 时 runs 的 FK 是 `SET NULL`（历史必须保留）。

## 决策

1. **全局归属**：`eval_datasets` 不加 workspace/KB 外键，任何登录用户可 CRUD（路由仅 `withAuth`，无新 guard）；跑 eval 仍需对目标 KB 通过 `requireKnowledgeBaseAccess`。同一 goldset 天然可以对多个 KB 复用（它绑定的是"逻辑文档集"，不是某个 KB 实例）。
2. **revision 乐观并发 + hash 内容标识**：并发令牌是 `eval_datasets.revision`（迁移 015 的整数列）——除创建外的所有写操作携带 `expectedRevision`，在 `SELECT … FOR UPDATE` 行锁内校验并在任何写（含改名/描述）后 +1，过期 → 409 `dataset_changed`。`dataset_hash`（`lib/eval/hash.ts` 的排序键 canonical SHA-1）只表示 case 内容，每次写后重算写回，仅服务 run 可比性。初版曾让 hash 一体两用充当并发令牌，review 指出元数据变更不改 hash、并发改名会静默覆盖（旧页面甚至能用同一 hash 删掉被改名的集合），遂拆分为双字段。上限 `MAX_GOLDSET_CASES = 50`（`lib/validation.ts`）在创建 / 单条新增 / 批量导入三处强制；`(dataset_id, case_key)` 唯一索引由迁移 014 兜底。
3. **run 一致性只看 hash**：`/api/eval/run` 一次快照贯穿（gate → run → 落库），`dataset_name` / `dataset_hash` 取自快照；dataset 被删后 run 以 `dataset_id = NULL` 保存。两个 run 可比 ⇔ `dataset_hash` 相等（`canCompare`，`lib/eval/goldset.ts`），不要求 `dataset_id` 相同——孤儿 run 之间仍可互比。

## 备选方案

### KB 级归属（dataset 挂 knowledge_base_id）

优点：权限模型与 runs 完全一致，preflight 永远针对"自己的" KB。
缺点：goldset 与语料是多对多关系——同一套 olympus 用例要对 demo KB、克隆 KB、重嵌入后的 KB 反复评测；挂 KB 意味着复制用例集，hash 相同却是不同行，对比语义反而混乱。

### Workspace 级归属

优点：符合现有租户模型，可防跨团队读取。
缺点：本应用的评测集不含敏感数据（问题 + 关键词 + 文件名），跨 workspace 复用是真实需求（demo/个人双 workspace 同评测）；引入 guard、迁移和 UI 归属选择器的成本换不来实际隔离收益。需要时可后补 `workspace_id` 列做增量收紧。

### 版本表（快照表，每次编辑生成新版本行）

优点：可回溯任意历史版本，run 指向精确版本行。
缺点：`eval_runs` 已经持有 name/hash 快照，回溯"跑过的内容"已经足够；版本表引入 GC、版本选择 UI 与"当前版本"指针维护，是为 50 条上限的小表设计的重机制。hash 即版本标识，等价且零额外 schema。

## 取舍

优点：
+ 零新 guard、零新租户迁移；管理面十分钟能讲清楚。
+ 并发与可比性各归一个字段（revision / dataset_hash），语义不重叠、互不牵连。
+ 孤儿 run 语义干净：删除集合不销毁历史，同 hash 仍可互比。

缺点：
− 任何登录用户可改/删任何 goldset（含内置集）——单团队自部署可接受，多租户 SaaS 化前需收紧。
− 无版本回溯：编辑即前进，只能靠 run 快照追溯跑过的内容。
− hash 对字段顺序以外的一切敏感：mapper（`lib/eval/goldset.ts`）必须保持"缺省字段省略"的 canonical 形状，有单测钉住。

## 影响

- 新增数据面 `lib/db/eval-datasets.ts` 与迁移 `db/migrations/014_managed_eval_datasets.sql`；`lib/eval/dataset.ts` 降级为 seed 模板（`pnpm seed:demo` 仅在同名缺失时创建）。
- 所有 eval 入口（`/api/eval/run`、`/api/eval/validate`、`pnpm eval:hybrid-ab`）以 `datasetId` 为参数，跑前执行同一套两层校验（structural lint + KB preflight，`lib/eval/validate.ts`）。
- UI 对比（baseline / compare tab）以 `canCompare` 为唯一判据；`datasetId === null` 标"数据集已删除"，hash 与当前集合不同标"数据集已变更"。
