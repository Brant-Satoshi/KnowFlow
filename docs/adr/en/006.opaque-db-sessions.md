# ADR-006: Opaque database sessions instead of JWT

## Status
Accepted

## Context
The app needs authenticated sessions across pages, API routes, and the SSE
chat stream. Requirements: immediate revocability (a removed workspace
member must lose access on the next request; ADR-007/008 depend on this),
simplicity, and no third-party auth dependency.

## Decision
- Sessions are opaque 256-bit tokens (`randomBytes(32).toString('base64url')`)
  stored as rows in a `sessions` table with a 30-day expiry
  (`lib/auth/sessions.ts`)
- Every API route validates against the DB via `requireUser()`; the edge
  proxy (`proxy.ts`) only checks cookie **presence** for page navigation
  redirects. There is no DB access on the edge runtime.
- Logout and expiry are row deletion/filtering; revocation is instant

## Alternatives considered

### JWT (stateless)
Pros:
- No DB read per request; horizontally free

Cons:
- Revocation requires denylists or short TTL + refresh machinery. That is
  exactly the property this app can't compromise on (instant membership
  revocation).
- Claims (e.g. workspace roles) go stale inside the token lifetime

### NextAuth / Auth.js
Pros:
- OAuth providers, battle-tested cookie handling

Cons:
- Large dependency for email+password only; session model would still need
  the DB strategy to get revocability

## Tradeoffs

Pros:
+ Instant, trivially correct revocation and logout
+ No token-parsing attack surface; the token carries zero information
+ Same freshness guarantee as the authz guards: every request re-checks

Cons:
- One indexed DB read per authenticated request (piggybacks on the pooled
  Postgres connection the request needs anyway)
- Edge redirect is cosmetic-only: a forged cookie passes the proxy but dies
  at the first `requireUser()`; must never treat proxy passage as authn

## Consequences
- Authorization data (workspace roles) is *never* cached in the session.
  `lib/authz/access.ts` joins live tables per request, which is what makes
  member removal take effect immediately.
- Session cleanup is lazy (expiry filter in the lookup join); a periodic
  purge is optional hygiene, not correctness

## Notes
`getSessionUser` never selects the password hash; `AuthUser` is `{id, email}`.
