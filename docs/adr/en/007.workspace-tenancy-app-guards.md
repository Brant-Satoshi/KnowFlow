# ADR-007: Shared-schema workspace tenancy with app-layer guards

## Status
Accepted

## Context
The app is multi-tenant: users must never see another workspace's knowledge
bases, files, chunks, conversations, or eval runs. The RAG pipeline makes
this sharp: a vector search that ignores tenancy leaks document content
into answers. Tenancy also had to be retrofit onto an existing single-user
schema (migration 010 backfill).

## Decision
Row-level tenancy in one shared schema, enforced in the application layer:

- `users -> workspaces -> workspace_members` (roles owner/admin/member);
  `knowledge_bases.workspace_id NOT NULL` is the tenant anchor; all other
  data inherits tenancy through FK chains (files->KB, chunks->file,
  conversations->KB, eval_runs->KB)
- Centralized guards in `lib/authz/access.ts`
  (`require{KnowledgeBase,Conversation,File,EvalRun,WorkspaceRole}Access`),
  all resolving through a `knowledge_bases x workspace_members` join
- Error semantics: cross-tenant / non-member -> **404**
  (`NotFoundOrForbiddenError`, anti-enumeration); authenticated member lacking
  a role -> **403** with a machine-readable `code`
- Vector search takes the KB filter inside the SQL (`searchChunks`), so no
  unscoped retrieval path exists

## Alternatives considered

### Postgres Row-Level Security (RLS)
Pros:
- Enforcement at the lowest layer; impossible to forget in a new route

Cons:
- Needs per-request `SET`/JWT context, which fights connection pooling
  (Supabase transaction pooler) and Drizzle's pooled client
- Policies on FK-chained tables (chunks via files via KBs) get complex fast

### Schema-per-tenant / database-per-tenant
Pros:
- Hard isolation; per-tenant backup/restore

Cons:
- Migration fan-out and connection management for what is a
  many-small-tenants product shape

### `user_id` column ownership (no workspaces)
Pros:
- Simplest possible filter

Cons:
- No collaboration story at all; retrofitting sharing later means this
  exact migration anyway

## Tradeoffs

Pros:
+ Guards are explicit, greppable, and unit-testable; one file to audit
+ Membership joins are evaluated per request -> revocation is immediate (with
  ADR-006 sessions)
+ Joining a workspace grants full RAG capability with zero extra code.
  Guards *are* the membership check.

Cons:
- Every new route must remember to call a guard. Discipline is enforced by
  review and the E2E matrix (cross-tenant->404 / own->200 / anon->401), not
  by the database.
- 404-for-existing-resources makes debugging slightly less obvious (by design)

## Consequences
- The single-owner invariant is defense-in-depth: app predicates
  (`role != 'owner'`) **plus** a partial unique index
  (`workspace_members_single_owner_idx`, migration 011)
- Conversations have no `user_id`: within a workspace, KB conversations are
  shared among members. This is a documented product behavior, not an oversight.

## Notes
Ownership transfer, workspace rename/delete UI, and per-user private
conversations are intentionally deferred.
