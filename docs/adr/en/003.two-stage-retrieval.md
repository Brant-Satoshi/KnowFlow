# ADR-003: Two-stage retrieval: vector recall then rerank

## Status
Accepted

## Context
Single-stage vector search returns semantically *similar* chunks, but cosine
similarity over bi-encoder embeddings is a blunt relevance signal: it misses
keyword-exact matches and ranks near-duplicates above complementary context.
Prompt budget only allows a handful of chunks, so ordering quality directly
drives answer quality and citation precision.

## Decision
Retrieve wide, then rerank narrow (per chat request in
`app/api/chat/stream/route.ts`):

1. Vector recall: top-20 chunks under a max cosine distance (`searchChunks`)
2. Rerank recall set with a cross-encoder: `cohere/rerank-v3.5` via
   OpenRouter (`lib/rag/rerank.ts`)
3. Take the top 5 into the prompt; each carries `_rerankScore` (or the
   vector `_distance` fallback) surfaced in the citation UI

`RERANK_ENABLED=false` degrades gracefully to pure vector order.

## Alternatives considered

### Single-stage vector top-K
Pros:
- One fewer network hop, lower latency and cost

Cons:
- Measurably worse ordering (the built-in eval page exists to quantify this:
  recall/precision/nDCG/MRR with and without rerank)

### Hybrid retrieval (BM25 + vector fusion)
Pros:
- Catches keyword-exact queries embeddings miss

Cons:
- Needs a text-search index and score fusion tuning; deferred until eval
  data shows the gap matters (later landed in ADR-010 as a default-off,
  toggleable capability — the eval shows no gain on the current dataset)

### Self-hosted cross-encoder
Pros:
- No per-call API cost

Cons:
- GPU serving infrastructure for a portfolio-scale app

## Tradeoffs

Pros:
+ Precision of a cross-encoder with the recall of ANN
+ Feature-flagged: rerank outage never breaks chat
+ Scores are explainable in the UI (rerank vs vector badge)

Cons:
- Adds one external API call of latency per message (mitigated by the
  `progress` SSE stage so the user sees "reranking")
- Rerank cost scales with recall width (20 docs/query)

## Consequences
- Retrieval quality is measurable, not assumed: `/eval` persists runs with
  `use_rerank` as a variable
- Tenancy is enforced *before* this stage (recall is already KB-scoped), so
  rerank/citations never see cross-tenant data

## Notes
Recall width (20), rerank cut (8), and prompt cut (5) are code constants.
Tune against eval runs, not intuition.
