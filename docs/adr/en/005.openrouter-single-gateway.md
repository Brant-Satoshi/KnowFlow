# ADR-005: OpenRouter as the single model gateway

## Status
Accepted

## Context
The app needs three model capabilities: chat completion (streaming),
embeddings, and reranking. Ideally this avoids managing three vendor accounts,
three SDKs, and three billing relationships.

## Decision
Route all three through OpenRouter with one API key:

- Chat: streaming Chat Completions; per-conversation model choice from a
  curated catalog (`lib/llm/catalog.ts`), persisted on `conversations.model`
- Embeddings: `OPENROUTER_EMBEDDING_MODEL` (default `text-embedding-3-small`)
- Rerank: `cohere/rerank-v3.5`

No vendor SDKs: plain `fetch` against the OpenAI-compatible surface.

## Alternatives considered

### Direct vendor APIs (OpenAI + Cohere + ...)
Pros:
- One less hop; vendor-native features and error semantics

Cons:
- Multiple keys/accounts; model switching becomes code changes per vendor

### Vercel AI SDK / LangChain abstraction layer
Pros:
- Provider abstraction, built-in streaming helpers

Cons:
- Heavy dependency for a pipeline this explicit; hides the SSE protocol this
  app deliberately owns (ADR-001)

## Tradeoffs

Pros:
+ One key, one bill, one base URL; models are catalog entries, not code
+ Per-conversation model switching is a product feature for free
+ Easy A/B of chat models against the eval page

Cons:
- Extra proxy hop on every call (latency, availability coupling)
- Error bodies are double-wrapped; `extractUpstreamMessage`
  (`lib/llm/chat.ts`) exists solely to unwrap provider errors from
  OpenRouter's envelope
- Embedding model choice is constrained by the fixed 1536-dim column
  (ADR-002)

## Consequences
- `resolveChatProvider` centralizes url/key/model resolution; adding a model
  is a catalog entry + nothing else
- Rate limits and outages are observed through one provider's status, which
  simplifies ops but concentrates risk

## Notes
`RERANK_ENABLED=false` (ADR-003) is the standing mitigation for
rerank-specific upstream failures.
