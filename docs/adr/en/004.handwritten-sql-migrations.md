# ADR-004: Handwritten SQL migrations; Drizzle for types only

## Status
Accepted

## Context
The schema uses features ORMs generate poorly or not at all: `vector(1536)`
columns, HNSW indexes, partial unique indexes (single-owner invariant),
`DO $$` backfill blocks, and named CHECK constraints. Migrations also run
against two targets (local Docker Postgres and Supabase) with different
connection mechanics.

## Decision
- Migrations are handwritten, numbered SQL files in `db/migrations/NNN_*.sql`,
  the single source of truth for the schema
- Every file is idempotent (`IF NOT EXISTS`, guarded `DO` blocks) so both
  Makefile targets (`migrate` via docker exec, `migrate-supabase` via psql)
  can re-run the full sequence safely
- Drizzle models in `lib/db/schema/*.ts` exist **only** for ORM type
  inference; `drizzle-kit` never generates migrations (`drizzle.config.ts`
  points generated output at a scratch dir)

## Alternatives considered

### drizzle-kit generate
Pros:
- Schema and migration can't drift; less typing

Cons:
- Poor support for pgvector/HNSW/partial indexes/backfills. Every real
  migration would need hand-editing anyway.
- Generated SQL is harder to review than intent-written SQL

### A migration framework (Flyway/dbmate/node-pg-migrate)
Pros:
- Version tracking table, up/down, checksums

Cons:
- New dependency for what `make` + idempotent SQL already covers at this scale

## Tradeoffs

Pros:
+ Full control and reviewability of DDL, including backfills
+ Idempotency means "apply everything" is the only operation to know
+ Works identically against Docker and Supabase

Cons:
- **Dual maintenance**: `schema/*.ts` can drift from SQL; nothing enforces
  parity except review discipline (both files change in the same PR)
- No down-migrations; rollback = forward-fix
- No applied-version tracking; idempotency substitutes for it

## Consequences
- Adding a table is a 4-step recipe (schema.ts, NNN.sql, Makefile x 2
  targets, query module), documented in CLAUDE.md
- DDL against Supabase should use the session pooler (:5432)/direct
  connection, not the transaction pooler (:6543)

## Notes
The Drizzle models still pay for themselves: typed query builders and
`$inferSelect` types across `lib/db/*`.
