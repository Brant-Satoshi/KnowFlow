# ADR-001: Use SSE + buffer + flush for streaming

## Status
Accepted

## Context
The system needs to stream LLM responses to the frontend in real time.

We evaluated different streaming approaches to improve user experience:
- WebSocket
- Server-Sent Events (SSE)

Additionally, raw token streaming from LLM providers can cause:
- UI jitter
- Too frequent re-renders
- Poor readability

## Decision
We choose:
- Server-Sent Events (SSE) for streaming transport
- Buffer + flush mechanism on the frontend

Streaming strategy:
1. Backend sends incremental chunks via SSE
2. Frontend accumulates chunks into a buffer
3. Use requestAnimationFrame to flush updates to UI

## Alternatives considered

### WebSocket
Pros:
- Full duplex communication

Cons:
- Overkill for one-way streaming
- More complex infra
- Harder to debug

### Direct token rendering (no buffer)
Pros:
- Simpler implementation

Cons:
- UI flickering
- Too many renders
- Bad UX

## Tradeoffs

Pros:
+ Simple and reliable
+ Works well with HTTP
+ Good compatibility (Edge / Serverless)
+ Smooth UI with buffer + flush

Cons:
- One-way communication only
- Needs manual buffer control on frontend

## Consequences

- Must implement buffer logic:
  - streamBufferRef
  - requestAnimationFrame
- Need SSE event protocol (meta / token / done)
- Frontend controls rendering frequency

## Notes

Frontend key pattern:

- streamBufferRef: accumulate tokens
- flushRafRef: schedule UI updates
- streamingAssistantIdRef: track active message

This ensures:
- smooth streaming
- reduced render cost
