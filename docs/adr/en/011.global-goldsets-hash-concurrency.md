# ADR-011: Global goldsets + revision-based optimistic concurrency (hash as content identity)

## Status

Accepted

## Context

Eval golden sets were hardcoded in `lib/eval/dataset.ts` and could only be extended by editing code; the `eval_datasets` / `eval_cases` tables existed (006) but held only code snapshots written by the runtime `ensureDataset`. Making goldsets user-manageable on `/eval` (manual form + JSON batch import) forces two decisions: which tenancy level a goldset belongs to, and the consistency model for concurrent edits and run comparability.

Constraints: eval runs themselves are KB-scoped (`requireKnowledgeBaseAccess`); `eval_runs` already carries snapshot columns `dataset_name` / `dataset_hash` plus `eval_runs_hash_idx`; deleting a dataset `SET NULL`s the runs' FK (history must survive).

## Decision

1. **Global ownership**: `eval_datasets` gets no workspace/KB foreign key; any signed-in user may CRUD (routes use plain `withAuth`, no new guard); running an eval still requires `requireKnowledgeBaseAccess` on the target KB. A goldset is bound to a logical document set, not a KB instance, so reusing one across KBs is the normal case.
2. **Revision-based optimistic concurrency + hash as content identity**: the concurrency token is `eval_datasets.revision` (integer, migration 015) â€” every write except creation carries `expectedRevision`, checked under a `SELECT â€¦ FOR UPDATE` row lock and bumped on every write, metadata included (stale â†’ 409 `dataset_changed`). `dataset_hash` (sorted-key canonical SHA-1, `lib/eval/hash.ts`) represents case content only, recomputed after each write, and serves run comparability exclusively. The first design used the hash as the token too; review showed metadata writes never change the hash, so concurrent renames could silently overwrite each other (and a stale page could delete a renamed dataset) â€” hence the split. The `MAX_GOLDSET_CASES = 50` cap (`lib/validation.ts`) is enforced at create, single add, and batch import; a unique `(dataset_id, case_key)` index (migration 014) backstops the app checks.
3. **Run comparability is hash-only**: `/api/eval/run` reads one snapshot for gate + run + persistence; `dataset_name` / `dataset_hash` always come from that snapshot; a dataset deleted mid-run leaves the run saved with `dataset_id = NULL`. Two runs are comparable â‡” their `dataset_hash` values match (`canCompare`, `lib/eval/goldset.ts`) â€” equal `dataset_id` is deliberately not required, so orphaned runs stay mutually comparable.

## Alternatives

### KB-level ownership (dataset carries knowledge_base_id)

Pros: permission model identical to runs; preflight always targets "its own" KB.
Cons: goldsets and corpora are many-to-many â€” the same olympus set gets evaluated against the demo KB, clones, and re-embedded KBs. KB ownership forces duplicating case sets that hash identically yet are different rows, muddying comparison semantics.

### Workspace-level ownership

Pros: matches the existing tenancy model; prevents cross-team reads.
Cons: goldsets here contain no sensitive data (questions + keywords + file names), and cross-workspace reuse is a real need. A guard, a migration, and an ownership picker buy no actual isolation value; a `workspace_id` column can be added later as an incremental tightening if needed.

### Version table (snapshot row per edit)

Pros: arbitrary historical versions recoverable; runs can point at exact version rows.
Cons: `eval_runs` already snapshots name/hash, which suffices to know what ran; a version table adds GC, a version-picker UI, and current-pointer maintenance â€” heavy machinery for a 50-case-capped table. The hash *is* the version identifier, at zero extra schema.

## Trade-offs

Pros:
+ Zero new guards, zero tenancy migrations; the management surface stays explainable in minutes.
+ Concurrency and comparability each own one field (revision / dataset_hash) with non-overlapping semantics.
+ Clean orphan semantics: deleting a set never destroys history, and equal-hash runs stay comparable.

Cons:
âˆ’ Any signed-in user can edit/delete any goldset (built-ins included) â€” fine for single-team self-hosting, must be tightened before multi-tenant SaaS.
âˆ’ No version rollback: edits move forward; only run snapshots witness what ran.
âˆ’ The hash is sensitive to everything but key order: the mapper (`lib/eval/goldset.ts`) must keep its canonical omit-absent-optionals shape, pinned by unit tests.

## Consequences

- New data layer `lib/db/eval-datasets.ts` and migration `db/migrations/014_managed_eval_datasets.sql`; `lib/eval/dataset.ts` is demoted to seed templates (`pnpm seed:demo` creates built-ins only when the name is absent).
- Every eval entry point (`/api/eval/run`, `/api/eval/validate`, `pnpm eval:hybrid-ab`) takes a `datasetId` and runs the same two-layer gate (structural lint + KB preflight, `lib/eval/validate.ts`) before executing.
- Comparison UI (baseline / compare tab) uses `canCompare` as the single criterion; `datasetId === null` is labeled "dataset deleted", a hash differing from the current set "dataset changed".
