# ADR-009: Hand-rolled typed i18n

## Status
Accepted

## Context
The UI ships in English and Chinese. Every user-visible string, including
`aria-label`s, must exist in both. The app is a client-heavy SPA-style App
Router app where language preference lives in `localStorage`.

## Decision
A single typed dictionary, no i18n library:

- `lib/i18n/translations.ts`: one object, `en`/`zh` x sections
  (`home`/`chat`/`eval`/`auth`); `useLanguage()` context selects the locale
- Type safety via inference: components receive
  `t: ReturnType<typeof useLanguage>["home"]`, so a missing key is a
  compile error at the call site
- Parameterization by convention: `"No results for {query}"` +
  `.replace("{query}", ...)`
- Sub-components take `t` as a prop instead of calling the hook, keeping
  render ownership explicit
- **Data vs UI boundary**: user data is never translated, but
  *system-generated* names are localized at display time
  (`lib/i18n/workspace-name.ts` maps "My Workspace" / "<x>'s Workspace"
  patterns; genuine user-chosen names pass through)

## Alternatives considered

### next-intl / react-i18next
Pros:
- ICU plurals, number/date formatting, lazy per-locale bundles,
  extraction tooling

Cons:
- A dependency and runtime for two locales and a few hundred keys;
  string keys are untyped by default (typing them back costs setup)

### Server-side locale (cookie + RSC)
Pros:
- No flash of wrong language; server can localize generated content

Cons:
- This app renders chat UI client-side anyway; localStorage + context is
  simpler and matches the existing theme toggle

## Tradeoffs

Pros:
+ Zero dependencies; keys are compile-time checked end to end
+ Adding a language = adding one object literal
+ Greppable: every string literal lives in one file

Cons:
- No pluralization/ICU. Count-sensitive strings need manual phrasing.
- Both locales must be edited together; nothing but review + types catches
  a key added to `en` only (the type intersection does flag shape drift)
- Server code can't localize (locale is client state), hence the
  display-layer approach for server-generated names

## Consequences
- Backend-generated user-facing text (default workspace names, default KB
  name) should stay within known patterns so the display layer can localize
  them; free-form server strings would leak English into the zh UI
- Error *messages* from the API are English; the client maps known `code`s
  to translated strings where UX matters

## Notes
`components/settings-menu.tsx` predates this convention (inline labels);
legacy exception, do not copy.
