# ADR-002: Store vectors in Postgres with pgvector

## Status
Accepted

## Context
The RAG pipeline needs a vector store for chunk embeddings (1536-dim) with
cosine similarity search. The same request path also needs relational data:
files, knowledge bases, conversations, and (since ADR-007) workspace
membership for tenant filtering.

## Decision
Keep everything in one PostgreSQL database (Supabase-hosted) using the
`pgvector` extension:

- `chunks.embedding vector(1536)` with an HNSW index for cosine distance
- Similarity search is plain SQL (`embedding <=> $1::vector`), so tenant and
  knowledge-base scoping are `JOIN`/`WHERE` clauses in the same query
- One backup/migration/connection story for all data

## Alternatives considered

### Dedicated vector DB (Pinecone / Qdrant / Milvus)
Pros:
- Purpose-built ANN performance and metadata filtering at large scale
- Namespace-per-tenant isolation primitives

Cons:
- Second datastore to operate, secure, and keep consistent with Postgres
- Chunk metadata would be duplicated or fetched via a second round trip
- Overkill for the current corpus size

### Postgres without ANN index (exact scan)
Pros:
- No index build cost, exact results

Cons:
- O(n) per query; latency grows linearly with corpus

## Tradeoffs

Pros:
+ Transactional consistency between chunks, files, and KBs (FK cascades work)
+ Tenant filtering is a join, not an application-side merge
+ Zero extra infrastructure

Cons:
- The HNSW index is global; per-KB/tenant scoping is a post-filter on ANN
  results. With many tenants and skewed corpus sizes, recall for small
  tenants can degrade and `topK` may need over-fetching
- Embedding dimension (1536) is baked into the schema; switching embedding
  models with a different dimension is a migration, not a config change

## Consequences
- `searchChunks` (`lib/db/chunks.ts`) is handwritten SQL, not ORM-generated
- Scaling path if needed: partial indexes, IVFFlat per shard, or extracting
  a vector service later. The SQL boundary makes that swap localized.

## Notes
`OPENROUTER_EMBEDDING_DIMENSIONS` exists but must match the column type;
see CLAUDE.md warning.
