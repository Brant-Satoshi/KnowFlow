# ADR-010: Hybrid search (RRF fusion of vector + keyword), default off

## Status
Accepted

## Context
ADR-003 listed hybrid retrieval as an alternative "deferred until eval data
shows the gap matters." This branch built the missing pieces — first the
pg_trgm keyword recall leg (migration 013, `keywordSearchChunks`), then
Reciprocal Rank Fusion (RRF) to fuse it with the vector leg — so we can finally
answer the deferred question: **on this project's current corpus and dataset,
does fusing the keyword leg in actually improve retrieval quality?**

A decision is needed because "hybrid sounds better" is exactly the kind of
intuition ADR-003 refused to act on. The code now runs, so the built-in eval
must produce a measured verdict rather than defaulting it on and hoping.

## Decision
Land hybrid search but **default it off** (`HYBRID_SEARCH_ENABLED`, default
`false`):

1. RRF fusion is a pure function: `reciprocalRankFusion(lists, k=60)`
   (`lib/rag/fusion.ts`). It fuses on rank position alone, so the vector leg's
   cosine distances and the keyword leg's trigram similarities never need
   normalizing onto a shared scale; a chunk at 0-based rank `r` in a list
   contributes `1/(k+r+1)`, and a chunk both legs surface outranks one only a
   single leg found.
2. `recallChunks` (`lib/rag/retrieve.ts`) gains `mode: 'vector' | 'hybrid'`. In
   hybrid mode the keyword leg (a DB query) runs **concurrently** with the
   embedding network call, so fusion adds no latency on the critical path; both
   legs share the same KB scope and `RetrievalFilter`, so fusion can never mix
   tenants or bypass a filter; a keyword-leg failure degrades to vector-only
   recall.
3. `/api/rag/search` gains `mode: 'hybrid'` to preview fused recall with the
   same RRF.
4. Whether production chat uses hybrid is decided by `isHybridSearchEnabled()`
   (`lib/models.ts`), mirroring the `RERANK_ENABLED` flag shape.

**Why default off**: measured on the `olympus-zh` dataset against the "测试" KB
(67 chunks, with distractor files), hybrid vs vector-only:
- Without rerank (raw recall ordering): **worse** — recall@1 0.900→0.600,
  nDCG@3 0.854→0.708, MRR 0.900→0.750. On a small corpus with CJK trigrams and
  a low threshold (0.05), the keyword leg injects character-overlap noise that
  RRF floats above the genuinely most-relevant chunk.
- With rerank (production config): **a wash** — nearly every metric unchanged
  (precision@5 +0.020, nDCG@3 −0.013 — noise on 10 cases). The cross-encoder
  reorders the fused set, erasing both the harm and any gain.

With no measurable gain on this dataset, there is no reason to make every chat
request pay for an extra keyword DB query plus fusion. So vector-only stays the
production default; hybrid ships as a toggleable capability, to be re-measured
and potentially enabled once a dataset/corpus where keyword-exact matching
matters exists.

## Alternatives considered

### Default hybrid on
Pros:
- "Hybrid search" is literally this branch's name; intuitively the keyword leg
  catches exact matches embeddings miss

Cons:
- The eval shows zero-to-negative gain on the current dataset; adding a DB query
  + fusion latency per request for no measurable quality lift violates the very
  principle ADR-003 set — decide retrieval changes by eval, not intuition

### BM25 / tsvector full-text search instead of pg_trgm
Pros:
- BM25 is a more established lexical relevance signal with TF/IDF weighting

Cons:
- Postgres built-in FTS cannot tokenize continuous Chinese text; this is a
  mixed zh/en corpus, so that leg would be largely dead on Chinese queries (see
  migration 013's rationale). pg_trgm `word_similarity` is language-agnostic —
  a deliberate choice

### Weighted score fusion (normalized cosine + trigram similarity, weighted sum)
Pros:
- Could weight the vector leg higher to directly suppress keyword noise

Cons:
- Requires normalizing two differently-scaled scores and tuning weights
  per-corpus — exactly the "score fusion tuning" burden ADR-003 flagged. RRF is
  tuning-free and the right baseline to land first; weight tuning is what you do
  *after* the eval proves the baseline insufficient

### Don't land hybrid at all
Pros:
- Less code and configuration surface

Cons:
- The keyword leg is already merged (migration 013); only fusion was missing.
  Finishing it behind a flag lets a future keyword-heavy dataset flip the flag
  and re-measure instead of re-implementing. `/api/rag/search`'s
  `mode: 'hybrid'` is also a useful preview/debug entry point

## Tradeoffs

Pros:
+ The hybrid verdict is measured, not assumed — the decision carries a concrete
  eval delta table
+ Default off = production chat behaves exactly as before this branch, zero
  risk; the flag can be flipped any time
+ RRF is a pure, tuning-free function; both legs share scope/filter, so no
  cross-tenant or filter-bypass risk is introduced
+ The eval runner's new `retrievalMode` makes the vector-vs-hybrid comparison
  reproducible — swap the dataset and re-run

Cons:
- Ships a default-off code path (fusion + keyword leg + flag) that must be
  maintained
- The verdict is bounded by a single dataset (`olympus-zh`, small and
  embedding-friendly); a dataset where keyword-exact matching matters more could
  reverse it — which is exactly the trigger to re-measure

## Consequences
- Production chat recall stays vector-only; `/eval` still measures the
  production pipeline (vector-only while the flag is off)
- Any proposal to "turn on hybrid" must first re-run the
  `retrievalMode: 'hybrid'` comparison on the target dataset and argue from the
  delta table, not reopen the intuition debate
- If a keyword-heavy dataset later proves the gap real, flipping the flag
  (`HYBRID_SEARCH_ENABLED=true`) is a config change, not a code change

## Notes
The measurement script was one-off and not committed;
`RunCuratedEvalOpts.retrievalMode` in `lib/eval/runner.ts` is the entry point to
reproduce the comparison. RRF's `k=60` and the keyword recall width (20) are
`RETRIEVAL` constants — tune there, not inline, if re-evaluating.
