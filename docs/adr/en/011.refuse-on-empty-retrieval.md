# ADR-011: Refuse in code when retrieval is empty; no rerank-score floor (default 0)

## Status
Accepted

## Context
Until now the chat pipeline called the LLM for every question — **including ones where
retrieval returned zero chunks**. Whether it then refused was left entirely to one line in
`buildQaPrompt`: "If the answer cannot be found in the context, say exactly ...".

That is a *request*, not a *guarantee*. An empty context is precisely where a model is most
tempted to fill the silence from its own weights, and precisely where answers the knowledge
base never contained get invented.

A decision was needed because "don't answer when you don't know" has two possible
implementations with very different reliability:

1. **Empty retrieval → refuse in code** (deterministic; does not depend on the model's good behavior)
2. **Low retrieval score → refuse in code** (requires a trustworthy relevance floor)

The second sounds like a natural extension of the first, and it was an explicit item on the
task list. But it only holds if **rerank relevance scores can distinguish "answerable" from
"unanswerable"**. That premise had to be measured, not assumed.

## Decision

### Adopted: refuse when retrieval comes back empty
`assessRetrieval()` (`lib/rag/refusal-gate.ts`) runs after `selectFinalChunks`. With no final
chunks, `emitRefusal()` (`lib/llm/refusal.ts`) streams the canned refusal as an ordinary turn
(`meta → progress → a single token → done`) and persists it, **without calling the LLM at all**.

- `meta.refusal` carries a machine-readable reason (`'empty' | 'low_score'`). It is the *only*
  signal that proves the gate fired: the refusal text proves nothing, since the prompt asks the
  LLM for that same sentence.
- The one exemption is a **conversation recap** (`isConversationSummaryQuery`), which is meant
  to be answered from history rather than retrieval. This required narrowing how `isSummaryQuery`
  is used: it is a loose substring match, and it previously routed "Summarize information about X"
  with zero recall into `buildConversationSummaryPrompt`, **discarding the user's question** and
  summarizing the chat instead — answering something they never asked.

### Rejected: a rerank-score floor (`RETRIEVAL.minRerankScore = 0`; mechanism kept, inert)

Measured with `pnpm eval:refusal` against `cohere/rerank-v3.5`: 7 negatives and 9 positives per
language, **EN calibrating, ZH held out**, each dataset run against a KB holding only its own
fixture (a bilingual KB lets the Chinese document answer the English negatives — we made exactly
that mistake; see Consequences).

| | lowest-scoring **answerable** question | highest-scoring **unanswerable** question |
| --- | ---: | ---: |
| olympus (EN) | 0.8808 ("which station detected the 41-hour pressure wave?") | **0.9055** ("which station recorded the Tharsis 18% dust intensification?") |
| olympus-zh (ZH) | 0.8212 | 0.8003 |

**On EN the distributions overlap, in the wrong direction**: the reranker scores a question the
corpus *cannot* answer above one it demonstrably *can*. The reason is not mysterious — a
cross-encoder measures **"is this chunk about the same thing as the query"**, and a near-miss
question is about exactly the same thing. It just asks for the one fact the document never states.
**Relevance is not answerability**, and the score does not carry the latter.

The calibration script *can* find a floor on EN with no added false refusals — **0.875**. It works
only because it lands **0.0058** below that 0.8808 answerable question. The minimum of nine
positives is a *sample* minimum, not the population's. That is not a calibrated threshold, it is a
lucky one.

**The held-out set proved it on the spot.** Taking EN's 0.875 to the ZH negatives it was never
chosen against: out-of-scope refusal 100%, but **in-scope false refusal 11.1%** — it refuses
`olympus-zh-relay-satellite` (0.8212), a question the document plainly answers. The threshold is
fitted to the EN sample, not to the reranker.

So: **the floor is built and tested, but ships at 0 (inert)**. It also disables itself wherever
it could not be trusted: rerank off, rerank degraded to recall order, the single-chunk
short-circuit (no score exists), or a reranker other than the calibrated one
(`OPENROUTER_RERANK_MODEL` ≠ `cohere/rerank-v3.5` — scores are not comparable across models).

**What catches the near-misses, then**: the prompt — and it measurably does. Running all 14
negatives from both datasets through the full pipeline (recall → rerank → gate → LLM): **14/14
refused, zero fabrications.** Four are caught by the code gate (empty recall); ten by the QA
prompt — including the nastiest near-miss at 0.8652 ("how many credits went to Phase 2?"), where
the model answers that the context does not say.

That is not luck. The model can *read the chunks* and see the fact isn't in them; code sees a
similarity scalar and cannot. The two mechanisms are complementary, not substitutes: code owns
"nothing was retrieved", the prompt owns "something was retrieved but the answer isn't in it".
Forcing a floor between them means deciding on a signal that does not carry the information.

## Alternatives

### Ship the 0.875 floor
Pros:
- Lifts out-of-scope refusal from 28.6% → 71.4% on the calibration set, with no added false
  refusal *on that set*

Cons:
- **The held-out set rejects it outright**: the same threshold refuses 11.1% of answerable
  questions in ZH
- Sits 0.0058 from a genuinely answerable question. Re-chunking, a new embedding model, or simply
  a different set of user questions starts refusing answerable ones — and a wrongly refused
  question is a harder failure to forgive than an occasional wrong answer
- The problem it exists to solve (fabrication on near-misses) **does not occur**: the prompt
  already refuses 14/14

### Use an LLM judge to decide whether the context suffices
Pros:
- The only signal that genuinely captures answerability

Cons:
- An extra LLM round-trip per request (latency + cost) to make a judgment the QA prompt is already
  making — paying twice for the same work
- If pursued, it should first be shown via eval that the QA prompt's near-miss refusal rate is
  actually inadequate

## Consequences
- An empty recall can no longer produce a fabricated answer, and skips an LLM call (refusals are
  instant).
- **Verify a negative against the whole fixture before authoring it.** The first draft of this ADR
  used "which station recorded the Tharsis 18%?" as a Chinese negative — and line 113 of
  `sample-zh.txt` says, in plain text, that Helios first detected the dust rise in eastern Tharsis.
  The model answered it correctly, with a citation, and was scored as a hallucination. **Grep the
  document; don't skim it.** Likewise, run each dataset against a KB holding only its own fixture:
  in a bilingual KB, the other language's document answers negatives this one cannot.
- `oosRefusalRate` / `inScopeFalseRefusalRate` (`lib/eval/runner.ts`) are how refusal correctness
  is measured. The existing `retrievalHit` / `citationHit` are **structurally incapable** of it:
  an out-of-scope case has no target chunks, so `retrievalHit` ("retrieved nothing relevant") is
  true whether the system refused or hallucinated, and `citationHit` ("cited nothing") passes an
  uncited fabrication exactly as happily as a refusal.
- Re-open the floor only by re-running `pnpm eval:refusal`; **any threshold is void after a
  reranker change**.
- Known boundary: when recall returns exactly one chunk, `rerankChunks` short-circuits and the
  chunk carries no score, so the floor is inert (recall's `maxDistance 0.6` remains the only
  guard). `olympus-oos-relay-latency` is exactly this case in the measured run.
