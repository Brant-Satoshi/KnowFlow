# ADR-002：使用 pgvector 将向量存储在 Postgres 中

## 状态
已接受

## 背景
RAG 管线需要一个向量存储，用于 chunk embedding（1536 维）和余弦相似度搜索。
同一条请求路径也需要关系型数据：文件、知识库、会话，以及（自 ADR-007 起）
用于租户过滤的 workspace membership。

## 决策
使用 `pgvector` 扩展，把所有数据保存在一个 PostgreSQL 数据库中
（Supabase 托管）：

- `chunks.embedding vector(1536)`，并为余弦距离建立 HNSW 索引
- 相似度搜索使用普通 SQL（`embedding <=> $1::vector`），因此租户和知识库范围
  可以作为同一个查询里的 `JOIN`/`WHERE` 条件
- 所有数据共享一套备份、迁移和连接方案

## 备选方案

### 专用向量数据库（Pinecone / Qdrant / Milvus）
优点：
- 面向大规模场景专门优化的 ANN 性能和元数据过滤能力
- 提供按租户 namespace 隔离的基础能力

缺点：
- 需要运维、保护并保持与 Postgres 一致的第二套数据存储
- chunk 元数据会被重复存储，或需要第二次往返查询
- 对当前语料规模来说过重

### 不使用 ANN 索引的 Postgres（精确扫描）
优点：
- 没有索引构建成本，结果精确

缺点：
- 每次查询都是 O(n)；延迟会随语料规模线性增长

## 取舍

优点：
+ chunk、文件和知识库之间保持事务一致性（外键级联可用）
+ 租户过滤是 join，而不是应用层合并
+ 不需要额外基础设施

缺点：
- HNSW 索引是全局的；按 KB/租户限定范围是在 ANN 结果上的后置过滤。
  当租户很多且语料规模分布不均时，小租户的召回可能下降，`topK` 可能需要超量获取
- embedding 维度（1536）被写入 schema；切换到不同维度的 embedding 模型需要迁移，
  不是配置变更

## 影响
- `searchChunks`（`lib/db/chunks.ts`）使用手写 SQL，而不是 ORM 生成
- 如需扩展，可选路径包括 partial index、按 shard 建 IVFFlat，或后续抽出向量服务；
  SQL 边界让这种替换保持局部化

## 备注
`OPENROUTER_EMBEDDING_DIMENSIONS` 存在，但必须与列类型匹配；见 CLAUDE.md
中的警告。
