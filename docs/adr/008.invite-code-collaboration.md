# ADR-008：用于 workspace 协作的邀请码

## 状态
已接受

## 背景
协作能力（ADR-007 的第二阶段）需要一种把 member 加入 workspace 的方式。
硬性约束包括：没有邮件发送基础设施（“零额外配置即可运行”）、不新增页面路由，
并且不能削弱多租户/防枚举保证。

## 决策
由 owner/admin 生成**邀请码**，接收者在“join workspace”对话框中粘贴：

- `workspace_invites`（migration 011）：128-bit `base64url` token，
  创建时固定角色（`member`/`admin`），有效期 1-168h（默认 72h），可撤销
- 过期前可多次使用（团队加入码），加入操作幂等
  （`onConflictDoNothing`；重复加入报告 `alreadyMember`）
- race-safe：加入时在事务内锁定 invite 行（`SELECT … FOR UPDATE`），因此已提交的撤销
  之后不可能再发生一次依赖被删除 invite 的加入
- 权限矩阵：admin 只能创建/查看/撤销 **member** invite；admin invite 只有 owner 可操作，
  对 admin 读起来是 404（不泄露存在性）；未知/过期 code → 通用 404
- token 明文存储；这有意采用与 session id（ADR-006）相同的威胁姿态，并记录在 schema 中

## 备选方案

### 邮件邀请
优点：
- 熟悉的同意流程，可直接面向未注册用户

缺点：
- 需要邮件 provider 依赖 + 模板 + deliverability 运维；违反零配置约束

### 通过 email 直接添加（无 code）
优点：
- 不需要新表；实现最小

缺点：
- 只适用于已注册用户；会在未同意的情况下添加用户；
  lookup 会泄露哪些 email 已有账号（枚举）

### 单次使用 code
优点：
- 每个被邀请者都有更严格的审计轨迹

缺点：
- 对常见的“分享到团队聊天”流程来说，每个队友一个 code 会增加摩擦；
  多次使用 + 短有效期 + 可撤销覆盖了相同的风险面

## 取舍

优点：
+ 零外部依赖；未注册用户也可使用（注册 → 粘贴 code）
+ 与 direct-add 不同，同意是内在的（接收者主动操作）
+ 按角色限定可见性，让 admin 级 token 不会落入 admin 手中

缺点：
- code 通过带外渠道传递（chat/DM）；拦截风险由过期和撤销约束，而不是绑定 email 地址
- 任何有数据库访问权限的人都能读取明文 token（在当前规模下接受；
  hash 会破坏“重新复制一个活跃 code”的功能）

## 影响
- 加入会立即授予对 workspace KB 的完整 RAG 访问，撤销会立即移除访问；
  两者来自 ADR-006 + ADR-007，而不是 invite 专属代码
- 对加入尝试不做 rate limit：token 不可猜（2^128），响应统一为 404

## 备注
UI 完全位于首页对话框中（switcher → manage members / join），遵守“不新增页面路由”的约束。
