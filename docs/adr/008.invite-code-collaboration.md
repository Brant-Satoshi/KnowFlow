# ADR-008: Invite codes for workspace collaboration

## Status
Accepted

## Context
Collaboration (phase 2 of ADR-007) needs a way to add members to a
workspace. Hard constraints: no email-sending infrastructure ("runnable
with zero extra setup"), no new page routes, and it must not weaken the
tenancy/anti-enumeration guarantees.

## Decision
Owner/admin-generated **invite codes**, pasted by the recipient in a
"join workspace" dialog:

- `workspace_invites` (migration 011): 128-bit `base64url` token, role fixed
  at mint (`member`/`admin`), expiry 1–168h (default 72h), revocable
- Multi-use until expiry (a team join code), join is idempotent
  (`onConflictDoNothing`; re-join reports `alreadyMember`)
- Race-safe: join locks the invite row (`SELECT … FOR UPDATE`) inside a
  transaction, so a committed revoke can never be followed by a join that
  relied on the deleted invite
- Permission matrix: admins mint/see/revoke **member** invites only; admin
  invites are owner-only and read as 404 to admins (no existence leak);
  unknown/expired codes → generic 404
- Tokens stored plaintext — deliberately the same threat posture as session
  ids (ADR-006), documented in the schema

## Alternatives considered

### Email invitations
Pros:
- Familiar consent flow, targets non-registered users directly

Cons:
- Requires an email provider dependency + templates + deliverability ops —
  violates the zero-setup constraint

### Direct add by email (no code)
Pros:
- No new table; smallest possible implementation

Cons:
- Only works for already-registered users; adds people without consent;
  the lookup leaks which emails have accounts (enumeration)

### Single-use codes
Pros:
- Tighter audit trail per invitee

Cons:
- One code per teammate is friction for the common "share to the team chat"
  flow; multi-use + short expiry + revocation covers the same risk surface

## Tradeoffs

Pros:
+ Zero external dependencies; works for not-yet-registered users
  (register → paste code)
+ Consent is inherent (recipient acts), unlike direct-add
+ Role-scoped visibility keeps admin-tier tokens out of admins' hands

Cons:
- Codes travel out-of-band (chat/DM) — interception risk bounded by expiry
  and revocation, not by binding to an email address
- Plaintext tokens readable by anyone with DB access (accepted at this
  scale; hashing would break the "re-copy an active code" feature)

## Consequences
- Joining instantly grants full RAG access to the workspace's KBs and
  revocation instantly removes it — both fall out of ADR-006 + ADR-007
  rather than invite-specific code
- No rate limiting on join attempts: tokens are unguessable (2^128) and the
  response is a uniform 404

## Notes
UI lives entirely in home-page dialogs (switcher → manage members / join),
respecting the "no new page routes" constraint.
