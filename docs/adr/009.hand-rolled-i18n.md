# ADR-009：手写带类型的 i18n

## 状态
已接受

## 背景
UI 提供英文和中文。每个用户可见字符串，包括 `aria-label`，都必须同时存在于两种语言中。
应用是偏客户端的 SPA 风格 App Router 应用，语言偏好保存在 `localStorage` 中。

## 决策
使用一个带类型的字典，不引入 i18n 库：

- `lib/i18n/translations.ts`：一个对象，包含 `en`/`zh` × section
  （`home`/`chat`/`eval`/`auth`）；`useLanguage()` context 选择 locale
- 通过推导获得类型安全：组件接收
  `t: ReturnType<typeof useLanguage>["home"]`，因此缺失 key 会在调用处变成编译错误
- 参数化使用约定：`"No results for {query}"` + `.replace("{query}", …)`
- 子组件通过 prop 接收 `t`，而不是自己调用 hook，让 render 所有权保持明确
- **数据 vs UI 边界**：用户数据永远不翻译，但*系统生成*名称会在展示时本地化
  （`lib/i18n/workspace-name.ts` 映射 "My Workspace" / "<x>'s Workspace" 模式；
  真正由用户选择的名称原样透传）

## 备选方案

### next-intl / react-i18next
优点：
- ICU 复数、数字/日期格式化、按 locale 懒加载 bundle、提取工具

缺点：
- 为两个 locale 和几百个 key 引入依赖和运行时；字符串 key 默认无类型
  （重新加类型需要额外配置）

### 服务端 locale（cookie + RSC）
优点：
- 不会闪现错误语言；服务端可以本地化生成内容

缺点：
- 本应用的 chat UI 本来就是客户端渲染；localStorage + context 更简单，
  并且与现有 theme toggle 匹配

## 取舍

优点：
+ 零依赖；key 从头到尾都有编译期检查
+ 添加语言 = 添加一个 object literal
+ 可搜索：所有字符串 literal 都在一个文件中

缺点：
- 没有复数/ICU；与数量相关的字符串需要手动措辞
- 两种 locale 必须一起编辑；除了 review + 类型外，没有别的机制捕捉只加到 `en` 的 key
  （type intersection 会标记形状漂移）
- 服务端代码无法本地化（locale 是客户端状态）；因此服务端生成名称采用展示层处理方式

## 影响
- 后端生成的用户可见文本（默认 workspace 名、默认 KB 名）应保持在已知模式中，
  这样展示层才能本地化；自由形式的服务端字符串会把英文泄露到中文 UI 中
- API 的错误 *message* 是英文；在 UX 重要的地方，客户端会把已知 `code` 映射到翻译字符串

## 备注
`components/settings-menu.tsx` 早于这个约定（内联 label）；这是历史例外，不要照抄。
