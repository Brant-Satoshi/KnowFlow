# ADR-001：使用 SSE + buffer + flush 实现流式输出

## 状态
已接受

## 背景
系统需要把 LLM 响应实时流式传输到前端。

我们评估了不同的流式方案，用来提升用户体验：
- WebSocket
- Server-Sent Events（SSE）

此外，LLM 提供商直接输出的原始 token 流可能导致：
- UI 抖动
- 重新渲染过于频繁
- 可读性较差

## 决策
我们选择：
- 使用 Server-Sent Events（SSE）作为流式传输通道
- 在前端使用 buffer + flush 机制

流式策略：
1. 后端通过 SSE 发送增量 chunk
2. 前端把 chunk 累积到 buffer 中
3. 使用 requestAnimationFrame 把更新 flush 到 UI

## 备选方案

### WebSocket
优点：
- 支持全双工通信

缺点：
- 对单向流式输出来说过重
- 基础设施更复杂
- 调试更困难

### 直接渲染 token（无 buffer）
优点：
- 实现更简单

缺点：
- UI 闪烁
- 渲染次数过多
- 用户体验差

## 取舍

优点：
+ 简单可靠
+ 与 HTTP 配合良好
+ 兼容性好（Edge / Serverless）
+ buffer + flush 能带来顺滑的 UI

缺点：
- 只支持单向通信
- 前端需要手动控制 buffer

## 影响

- 必须实现 buffer 逻辑：
  - streamBufferRef
  - requestAnimationFrame
- 需要 SSE 事件协议（meta / token / done）
- 由前端控制渲染频率

## 备注

前端关键模式：

- streamBufferRef：累积 token
- flushRafRef：调度 UI 更新
- streamingAssistantIdRef：跟踪当前活跃消息

这可以保证：
- 流式输出顺滑
- 降低渲染成本
