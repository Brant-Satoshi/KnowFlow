# ADR-004：手写 SQL migration；Drizzle 仅用于类型

## 状态
已接受

## 背景
schema 使用了一些 ORM 很难生成或完全无法生成的能力：`vector(1536)` 列、HNSW 索引、
partial unique index（单 owner 不变量）、`DO $$` 回填块，以及具名 CHECK 约束。
migration 还需要跑在两个目标上（本地 Docker Postgres 和 Supabase），且连接机制不同。

## 决策
- migration 是 `db/migrations/NNN_*.sql` 中手写、带编号的 SQL 文件，
  是 schema 的唯一事实来源
- 每个文件都是幂等的（`IF NOT EXISTS`、带 guard 的 `DO` block），因此两个
  Makefile target（通过 docker exec 的 `migrate`、通过 psql 的 `migrate-supabase`）
  都可以安全地重新执行完整序列
- `lib/db/schema/*.ts` 中的 Drizzle model **只**用于 ORM 类型推导；
  `drizzle-kit` 永远不生成 migration（`drizzle.config.ts` 把生成输出指向临时目录）

## 备选方案

### drizzle-kit generate
优点：
- schema 和 migration 不会漂移；少写一些代码

缺点：
- 对 pgvector/HNSW/partial index/回填支持较差；真实 migration 最终仍然需要手工编辑
- 生成的 SQL 比按意图手写的 SQL 更难 review

### migration 框架（Flyway/dbmate/node-pg-migrate）
优点：
- 版本跟踪表、up/down、checksum

缺点：
- 在当前规模下，`make` + 幂等 SQL 已经覆盖需求，引入框架只是新增依赖

## 取舍

优点：
+ 对 DDL 有完整控制权且易于 review，包括回填
+ 幂等性意味着只需要知道“应用全部”这一种操作
+ 对 Docker 和 Supabase 的行为一致

缺点：
- **双重维护**：`schema/*.ts` 可能与 SQL 漂移；除了 review 纪律外没有机制强制一致
  （两类文件应在同一个 PR 中同时修改）
- 没有 down-migration；回滚 = 向前修复
- 没有已应用版本跟踪；用幂等性替代

## 影响
- 新增表是 4 步流程（schema.ts、NNN.sql、Makefile × 2 个 target、query module）；
  已记录在 CLAUDE.md 中
- 针对 Supabase 执行 DDL 时应使用 session pooler（:5432）/direct connection，
  不应使用 transaction pooler（:6543）

## 备注
Drizzle model 仍然值得保留：它们为 `lib/db/*` 提供 typed query builder 和
`$inferSelect` 类型。
