# ADR-010：混合检索（RRF 融合向量 + 关键词），默认关闭

## 状态
已接受

## 背景
ADR-003 把混合检索列为一个"推迟到 eval 数据证明差距重要时再做"的备选方案。
本分支补齐了所需部件——先落地 pg_trgm 关键词召回腿（migration 013，
`keywordSearchChunks`），再实现 Reciprocal Rank Fusion（RRF）把它和向量腿融合——
于是我们终于可以回答那个被推迟的问题：**在本项目当前的语料和数据集上，把关键词腿
融合进来是否真的提升检索质量？**

需要一个决策，是因为"混合检索听起来更好"正是 ADR-003 拒绝凭直觉下判断的那类主张，
而现在代码已经能跑，必须用内置 eval 给出可度量的结论，而不是默认打开了事。

## 决策
落地混合检索，但**默认关闭**（`HYBRID_SEARCH_ENABLED`，默认 `false`）：

1. RRF 融合是纯函数：`reciprocalRankFusion(lists, k=60)`（`lib/rag/fusion.ts`）。
   只按排名位置融合，因此向量腿的余弦距离和关键词腿的 trigram 相似度**无需**归一到
   同一量纲；chunk 在某腿 0 基排名 `r` 处贡献 `1/(k+r+1)`，两腿都命中的 chunk 得分更高。
2. `recallChunks`（`lib/rag/retrieve.ts`）新增 `mode: 'vector' | 'hybrid'`。hybrid 模式下
   关键词腿（一次 DB 查询）与 embedding 网络调用**并发**发起，因此融合不在关键路径上增加延迟；
   两腿共用同一 KB scope 与 `RetrievalFilter`，融合不可能越权或绕过过滤；关键词腿失败时
   降级为纯向量召回。
3. `/api/rag/search` 新增 `mode: 'hybrid'`，用相同的 RRF 预览融合召回。
4. 生产 chat 是否走 hybrid 由 `isHybridSearchEnabled()`（`lib/models.ts`）决定，
   镜像 `RERANK_ENABLED` 的开关形态。

**为什么默认关闭**：在 `olympus-zh` 数据集、"测试" KB（67 chunk，含干扰文件）上度量，
hybrid 相对纯向量：
- 不启用 rerank（原始召回排序）：**变差**——recall@1 0.900→0.600，nDCG@3 0.854→0.708，
  MRR 0.900→0.750。小语料 + CJK trigram + 低阈值（0.05）让关键词腿引入字符重叠噪声，
  RRF 把这些 chunk 顶到真正最相关 chunk 之上。
- 启用 rerank（生产配置）：**基本持平**——各项指标几乎不变（precision@5 +0.020，
  nDCG@3 −0.013，在 10 条用例上属于噪声）。cross-encoder 重排融合集合，既抹掉了损害，
  也抹掉了任何收益。

既然在该数据集上没有可度量的收益，就没有理由让每个 chat 请求都多付一次关键词 DB 查询和
融合成本。因此纯向量保持为生产默认；混合检索作为可开关的能力落地，等到出现一个
关键词精确匹配更重要的数据集/语料再重新度量并考虑打开。

## 备选方案

### 默认打开混合检索
优点：
- "混合检索"正是本分支的名字；直觉上关键词腿能兜住 embedding 漏掉的精确匹配

缺点：
- eval 在现有数据集上显示零收益甚至负收益；给每个请求增加一次 DB 查询 + 融合延迟却换不到
  可度量的质量提升，违背 ADR-003 立下的"用 eval 而非直觉决定检索改动"的原则

### 用 BM25 / tsvector 全文检索代替 pg_trgm
优点：
- BM25 是更成熟的词法相关性信号，带词频/逆文档频率加权

缺点：
- Postgres 内置 FTS 无法切分中文连续文本；本项目是 zh/en 混合语料，这条腿会在中文查询上
  基本失效（见 migration 013 的说明）。pg_trgm 的 `word_similarity` 语言无关，是刻意的选择

### 加权分数融合（归一化 cosine + trigram 相似度按权重相加）
优点：
- 可以给向量腿更高权重，直接压制关键词噪声

缺点：
- 需要把两个不同量纲的分数归一并逐语料调权重——正是 ADR-003 担心的"分数融合调参"负担。
  RRF 无需调参，是先落地的正确基线；调权重属于"eval 证明基线不够"之后才做的事

### 干脆不落地混合检索
优点：
- 更少的代码和配置面

缺点：
- 关键词腿已经合入（migration 013），只差融合这一步；把它接完并放在开关后，未来出现
  关键词重的数据集时可以直接翻开关重测，而不必重新实现。`/api/rag/search` 的
  `mode: 'hybrid'` 也是一个有用的预览/调试入口

## 取舍

优点：
+ 混合检索的判断是可度量的，不是假设的——决策附带具体的 eval delta 表
+ 默认关闭 = 生产 chat 行为与本分支前完全一致，零风险；开关可随时打开
+ RRF 是纯函数、无需调参，两腿共用 scope/filter，不引入跨租户或越权风险
+ eval runner 新增的 `retrievalMode` 让"向量 vs 混合"对比可复现，换数据集即可重跑

缺点：
- 落地了一个默认不启用的代码路径（融合 + 关键词腿 + 开关），需要维护
- 结论受限于单一数据集（`olympus-zh`，语料小且对 embedding 友好）；换一个关键词精确匹配
  更重要的数据集，结论可能反转——这正是重新度量的触发条件

## 影响
- 生产 chat 召回保持纯向量；`/eval` 度量的仍是生产管线（开关关闭时即向量-only）
- 任何"打开混合检索"的提议都必须先在目标数据集上重跑 `retrievalMode: 'hybrid'` 对比，
  用 delta 表说话，而不是重开这场直觉之争
- 若未来一个关键词重的数据集证明差距成立，翻开关（`HYBRID_SEARCH_ENABLED=true`）即可，
  无需改代码

## 备注
度量脚本是一次性的，未入库；`lib/eval/runner.ts` 的 `RunCuratedEvalOpts.retrievalMode`
是复现对比的入口。RRF 的 `k=60` 与关键词召回宽度（20）是 `RETRIEVAL` 常量，若重新评估
应在那里调，而不是内联。
