# ADR-005：使用 OpenRouter 作为统一模型网关

## 状态
已接受

## 背景
应用需要三类模型能力：chat completion（流式）、embedding 和 reranking。
理想情况下，不希望同时管理三个供应商账号、三个 SDK 和三套账单关系。

## 决策
使用一个 API key，把三类能力全部通过 OpenRouter 路由：

- Chat：流式 Chat Completions；每个 conversation 可从 curated catalog
  （`lib/llm/catalog.ts`）中选择模型，并持久化到 `conversations.model`
- Embeddings：`OPENROUTER_EMBEDDING_MODEL`（默认 `text-embedding-3-small`）
- Rerank：`cohere/rerank-v3.5`

不使用供应商 SDK；直接用 `fetch` 调用兼容 OpenAI 的接口。

## 备选方案

### 直接调用供应商 API（OpenAI + Cohere + …）
优点：
- 少一次跳转；可获得供应商原生功能和错误语义

缺点：
- 多个 key/账号；按供应商切换模型会变成代码改动

### Vercel AI SDK / LangChain 抽象层
优点：
- provider 抽象，内置流式辅助能力

缺点：
- 对这条显式管线来说依赖过重；会隐藏本应用刻意自行拥有的 SSE 协议（ADR-001）

## 取舍

优点：
+ 一个 key、一个账单、一个 base URL；模型是 catalog entry，不是代码
+ 每个 conversation 切换模型天然成为产品功能
+ 可以很容易在 eval 页面上对 chat 模型做 A/B

缺点：
- 每次调用多一层代理跳转（延迟、可用性耦合）
- 错误体会被双重包装；`extractUpstreamMessage`（`lib/llm/chat.ts`）存在的唯一目的，
  就是从 OpenRouter envelope 中解出 provider 错误
- embedding 模型选择受固定 1536 维列约束（ADR-002）

## 影响
- `resolveChatProvider` 集中处理 url/key/model 解析；新增模型只需要增加 catalog entry
- rate limit 和故障都通过一个 provider 的状态观察，这简化了运维但也集中化了风险

## 备注
`RERANK_ENABLED=false`（ADR-003）是针对 rerank 上游故障的常设缓解手段。
