# ADR-006：使用不透明数据库 session，而不是 JWT

## 状态
已接受

## 背景
应用需要在页面、API 路由和 SSE chat stream 中使用认证 session。需求包括：
可立即撤销（被移除的 workspace member 必须在下一个请求失去访问权限；ADR-007/008
依赖这一点）、简单，以及不依赖第三方认证。

## 决策
- session 是不透明的 256-bit token（`randomBytes(32).toString('base64url')`），
  作为行存储在 `sessions` 表中，过期时间为 30 天（`lib/auth/sessions.ts`）
- 每个 API 路由都通过 `requireUser()` 到数据库校验；edge proxy（`proxy.ts`）
  只检查 cookie 是否**存在**，用于页面导航重定向；edge runtime 上不访问数据库
- 登出和过期都通过行删除/过滤实现；撤销即时生效

## 备选方案

### JWT（无状态）
优点：
- 每个请求无需读取数据库；水平扩展成本低

缺点：
- 撤销需要 denylist，或短 TTL + refresh 机制；而这正是本应用不能妥协的属性
  （成员资格必须即时撤销）
- claims（例如 workspace role）会在 token 生命周期内变旧

### NextAuth / Auth.js
优点：
- OAuth provider、经过充分验证的 cookie 处理

缺点：
- 对仅 email+password 的场景来说依赖过大；为了可撤销性，session model 仍然需要数据库策略

## 取舍

优点：
+ 撤销和登出即时生效且显然正确
+ 没有 token 解析攻击面；token 不携带任何信息
+ 与 authz guard 有相同的新鲜度保证：每个请求都重新检查

缺点：
- 每个已认证请求需要一次带索引的数据库读取（复用该请求本来就需要的 pooled Postgres 连接）
- Edge redirect 只是体验层优化：伪造 cookie 可以通过 proxy，但会在第一个 `requireUser()`
  处失败；绝不能把通过 proxy 视为已认证

## 影响
- 授权数据（workspace role）*永远不会*缓存在 session 中；`lib/authz/access.ts`
  每个请求都会 join 当前表，这正是移除 member 能即时生效的原因
- session 清理是惰性的（lookup join 中的过期过滤）；周期性清理只是可选卫生措施，
  不影响正确性

## 备注
`getSessionUser` 从不选择 password hash；`AuthUser` 是 `{id, email}`。
