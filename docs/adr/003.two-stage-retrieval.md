# ADR-003：两阶段检索：先向量召回，再 rerank

## 状态
已接受

## 背景
单阶段向量搜索会返回语义上*相似*的 chunk，但 bi-encoder embedding 上的余弦相似度
是比较粗糙的相关性信号：它会错过关键词精确匹配，也会把近重复内容排在互补上下文之前。
prompt 预算只允许放入少量 chunk，因此排序质量会直接影响回答质量和引用精度。

## 决策
先宽召回，再窄 rerank（每个 chat 请求在 `app/api/chat/stream/route.ts` 中执行）：

1. 向量召回：在最大余弦距离限制下取 top-20 chunk（`searchChunks`）
2. 使用 cross-encoder 对召回集合 rerank：通过 OpenRouter 调用
   `cohere/rerank-v3.5`（`lib/rag/rerank.ts`）
3. 取前 5 个放入 prompt；每个 chunk 都带有 `_rerankScore`（或回退到向量
   `_distance`），并在引用 UI 中展示

`RERANK_ENABLED=false` 时会平滑降级为纯向量排序。

## 备选方案

### 单阶段向量 top-K
优点：
- 少一次网络跳转，延迟和成本更低

缺点：
- 排序质量可测地更差（内置 eval 页面用于量化这一点：对比启用和不启用 rerank 时的
  recall/precision/nDCG/MRR）

### 混合检索（BM25 + 向量融合）
优点：
- 能捕获 embedding 遗漏的关键词精确查询

缺点：
- 需要文本搜索索引和分数融合调参；推迟到 eval 数据证明差距重要时再做
  （后续在 ADR-010 中落地为默认关闭的可开关能力——eval 显示当前数据集上无收益）

### 自托管 cross-encoder
优点：
- 没有按次 API 调用成本

缺点：
- 对 portfolio 规模的应用来说，需要 GPU serving 基础设施

## 取舍

优点：
+ 同时获得 cross-encoder 的精度和 ANN 的召回
+ 有特性开关：rerank 故障不会破坏 chat
+ 分数可在 UI 中解释（rerank vs vector badge）

缺点：
- 每条消息增加一次外部 API 调用延迟（通过 `progress` SSE 阶段缓解，让用户看到
  “reranking” 状态）
- rerank 成本随召回宽度增长（20 docs/query）

## 影响
- 检索质量是可度量的，而不是靠假设：`/eval` 会持久化带 `use_rerank` 变量的运行记录
- 多租户隔离在此阶段*之前*已经强制执行（召回已经限定到 KB），因此 rerank/引用永远不会看到跨租户数据

## 备注
召回宽度（20）、rerank 截断（8）和 prompt 截断（5）都是代码常量；应基于 eval
运行结果调参，而不是凭直觉。
