# ADR-007：共享 schema 的 workspace 多租户与应用层 guard

## 状态
已接受

## 背景
应用是多租户的：用户绝不能看到其他 workspace 的知识库、文件、chunk、conversation
或 eval run。RAG 管线让这个问题更加尖锐，因为忽略租户边界的向量搜索会把文档内容泄露到回答中。
多租户能力还必须被补到现有单用户 schema 上（migration 010 backfill）。

## 决策
在一个共享 schema 中做行级多租户，并由应用层强制执行：

- `users → workspaces → workspace_members`（角色为 owner/admin/member）；
  `knowledge_bases.workspace_id NOT NULL` 是租户锚点；其他数据通过外键链继承租户属性
  （files→KB、chunks→file、conversations→KB、eval_runs→KB）
- `lib/authz/access.ts` 中提供集中式 guard
  （`require{KnowledgeBase,Conversation,File,EvalRun,WorkspaceRole}Access`），
  全部通过 `knowledge_bases ⨝ workspace_members` join 解析
- 错误语义：跨租户 / 非 member → **404**（`NotFoundOrForbiddenError`，防枚举）；
  已认证 member 但缺少角色 → **403**，并返回机器可读的 `code`
- 向量搜索在 SQL 内部接收 KB filter（`searchChunks`），因此不存在未限定范围的检索路径

## 备选方案

### Postgres Row-Level Security（RLS）
优点：
- 在最低层强制执行；新路由不可能忘记

缺点：
- 需要每个请求设置 `SET`/JWT context，这会与连接池（Supabase transaction pooler）
  和 Drizzle 的 pooled client 冲突
- FK 链表上的 policy（chunks 经由 files 再经由 KBs）会很快变复杂

### 每租户一个 schema / 每租户一个数据库
优点：
- 强隔离；可按租户备份/恢复

缺点：
- 对“许多小租户”的产品形态来说，会带来 migration 扇出和连接管理成本

### `user_id` 列所有权（无 workspace）
优点：
- 最简单的过滤方式

缺点：
- 完全没有协作故事；以后补共享能力仍然会走这次迁移

## 取舍

优点：
+ guard 显式、可搜索、可单元测试；只需审计一个文件
+ membership join 每个请求都会执行 → 撤销即时生效（配合 ADR-006 session）
+ 加入 workspace 即获得完整 RAG 能力，无需额外代码；guard *就是* membership check

缺点：
- 每个新路由都必须记得调用 guard；这由 review 和 E2E 矩阵（cross-tenant→404 /
  own→200 / anon→401）保证，而不是由数据库保证
- 对已存在资源返回 404 会让调试稍微不直观（这是有意设计）

## 影响
- 单 owner 不变量是 defense-in-depth：应用谓词（`role != 'owner'`）**加上**
  partial unique index（`workspace_members_single_owner_idx`，migration 011）
- conversation 没有 `user_id`：在 workspace 内，KB conversation 会在 member 之间共享；
  这是记录在案的产品行为，不是疏漏

## 备注
owner 转移、workspace 重命名/删除 UI，以及按用户私有 conversation 都有意推迟。
